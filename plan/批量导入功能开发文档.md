## 一、功能概述

### 1.1 开发目标

完成组织管理模块的批量导入功能，实现基于**权限层级**的 Excel 批量导入与审核流程。

### 1.2 角色权限体系

```
┌─────────────────────────────────────────────────────────────────┐
│                        总部系统管理员                            │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐ │
│  │ 添加4S店信息管理员│  │ 添加总部信息员   │  │ 审核管理        │ │
│  │ (store_id=0)    │  │ (store_id=0)    │  │ 批量审核/启用   │ │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘ │
└─────────────────────────────────────────────────────────────────┘
           ↓                                    ↓
┌───────────────────────┐          ┌───────────────────────┐
│   4S店信息管理员       │          │     总部信息员         │
│ (由总部审核后启用)     │          │ (总部添加后直接启用)   │
│  ┌─────────────────┐  │          │  ┌─────────────────┐  │
│  │ 修改个人信息    │  │          │  │ 修改个人信息    │  │
│  │ (提交总部审核)  │  │          │  │ (提交总部审核)  │  │
│  └─────────────────┘  │          │  └─────────────────┘  │
│  ┌─────────────────┐  │          └───────────────────────┘
│  │ 添加业务顾问    │  │
│  │ 添加财务审核    │  │
│  │ 生成二维码     │  │
│  └─────────────────┘  │
└───────────────────────┘
```

### 1.3 权限矩阵

| 操作 | 总部系统管理员 | 4S店信息管理员 | 总部信息员 |
|------|--------------|--------------|----------|
| 添加4S店信息管理员 | ✅ | ❌ | ❌ |
| 添加总部信息员 | ✅ | ❌ | ❌ |
| 添加4S店业务顾问 | ❌ | ✅（仅本4S店）| ❌ |
| 添加4S店财务审核 | ❌ | ✅（仅本4S店）| ❌ |
| 修改个人信息 | ✅ | ✅（提交审核）| ✅（提交审核）|
| 审核他人信息 | ✅ | ❌ | ❌ |
| 批量审核/启用 | ✅ | ❌ | ❌ |
| 生成二维码 | ✅ | ✅（仅本人）| ❌ |
| 查看他店人员 | ✅ | ❌ | ❌ |

### 1.4 审核状态与启用流程

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│ 待审核  │ ──→ │ 已通过  │ ──→ │ 已启用  │
└─────────┘     └─────────┘     └─────────┘
     │               │               │
     ↓               ↓               ↓
  新增/修改       审核通过        手动启用
  默认状态       待手动启用       可登录使用
```

**说明：**
- `待审核`：新增或修改信息后的初始状态
- `已通过`：总部审核通过，但未启用
- `已启用`：可以登录系统使用

## 二、数据库表结构

### 2.1 店面表（store）- 新建

```sql
CREATE TABLE `store`(
  `store_id` int(11) NOT NULL AUTO_INCREMENT COMMENT '店面ID',
  `store_name` varchar(128) NOT NULL COMMENT '店面名称',
  `store_address` varchar(255) COMMENT '店面地址',
  `store_phone` varchar(16) COMMENT '联系电话',
  `store_manager` varchar(64) COMMENT '负责人',
  `examine_state` varchar(16) DEFAULT '已通过' COMMENT '审核状态：待审核/已通过/已拒绝',
  `user_id` int(11) DEFAULT '0' COMMENT '关联 user 表的 user_id',
  `create_by` int(11) DEFAULT '1' COMMENT '创建用户ID',
  `create_time` datetime DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `update_time` timestamp DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`store_id`),
  INDEX `idx_user_id` (`user_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='4S店/店面';
```

### 2.2 各角色表字段（含 store_id）

| 角色 | 表名 | store_id 关联 | 说明 |
|------|------|--------------|------|
| 信息员（总部） | `info_user` | `store_id=0` | 总部添加，直接启用 |
| 信息员（4S店） | `info_user` | `store_id>0` | 4S店信息管理员，需审核 |
| 业务顾问 | `business_consultant` | `store_id>0` | 由4S店信息管理员添加 |
| 财务审核 | `financial_user` | `store_id>0` | 由4S店信息管理员添加 |
| 会员 | `owner_user` | `store_id>0` | 关联4S店 |

### 2.3 store_id 字段说明

| store_id 值 | 含义 | 适用角色 |
|-------------|------|----------|
| 0 | 总部（不属于任何4S店）| 总部信息管理员、总部信息员 |
| > 0 | 具体的4S店ID | 4S店信息管理员、业务顾问、财务审核、会员 |

## 四、Excel 模板下载与上传接口

### 4.1 下载模板接口

```
GET /api/import/download_template
```

**请求参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| role_type | String | ✅ | 角色类型 |
| store_id | Integer | 部分 | 4S店ID（4S店管理员操作时必填）|

**role_type 枚举：**

| role_type | 说明 | 适用角色 | 下载时是否需要 store_id |
|-----------|------|----------|------------------------|
| `store_admin` | 4S店信息管理员 | 总部系统管理员 | ❌（自动设为目标店面）|
| `headquarter_info` | 总部信息员 | 总部系统管理员 | ❌（固定 store_id=0）|
| `consultant` | 业务顾问 | 4S店信息管理员 | ✅ |
| `financial` | 财务审核 | 4S店信息管理员 | ✅ |
| `member` | 会员 | 通用 | ✅（可选）|

### 4.2 上传导入接口

```
POST /api/import/excel_upload
Content-Type: multipart/form-data
```

**请求参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| file | File | ✅ | Excel 文件 |
| role_type | String | ✅ | 角色类型 |
| store_id | Integer | 部分 | 4S店ID（根据角色类型决定）|

**通用响应格式：**

```json
{
  "result": {
    "total": 100,
    "success": 95,
    "failed": 5,
    "pending_review": 95,
    "failed_list": [
      { "row": 3, "reason": "员工工号重复" },
      { "row": 7, "reason": "联系号码格式错误" },
      { "row": 15, "reason": "所属店面不存在" }
    ]
  }
}
```

**响应字段说明：**

| 字段 | 类型 | 说明 |
|------|------|------|
| total | Integer | 总行数 |
| success | Integer | 校验通过的行数 |
| failed | Integer | 校验失败的行数 |
| pending_review | Integer | 需要审核的行数（默认等于 success）|
| failed_list | Array | 失败详情列表 |

## 五、批量审核功能

### 5.1 批量审核接口

```
POST /api/review/batch_approve
Content-Type: application/json
```

**请求参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| role_type | String | ✅ | 角色类型 |
| ids | Array | ✅ | 待审核的记录ID数组 |
| action | String | ✅ | 审核动作：approve/reject |

**请求示例：**

```json
{
  "role_type": "store_admin",
  "ids": [1, 2, 3, 4, 5],
  "action": "approve"
}
```

**响应示例：**

```json
{
  "result": {
    "total": 5,
    "success": 5,
    "failed": 0,
    "failed_list": []
  }
}
```

### 5.2 批量启用接口

```
POST /api/review/batch_enable
Content-Type: application/json
```

**请求参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| role_type | String | ✅ | 角色类型 |
| ids | Array | ✅ | 待启用的记录ID数组 |

**业务逻辑：**
1. 仅 `examine_state = '已通过'` 的记录可以启用
2. 启用后将 `examine_state` 更新为 `已启用`
3. 启用后该用户可以登录系统

### 5.3 批量审核+启用（合并操作）

```
POST /api/review/batch_approve_and_enable
Content-Type: application/json
```

**请求参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| role_type | String | ✅ | 角色类型 |
| ids | Array | ✅ | 待审核并启用的记录ID数组 |

**业务逻辑：**
1. 将选中记录的 `examine_state` 从 `待审核` 直接变更为 `已启用`
2. 适用于批量导入后快速启用

## 六、审核流程详解

### 6.1 4S店信息管理员添加流程

```
总部系统管理员
     │
     │ 1. 批量导入4S店信息管理员
     │    └─→ role_type = 'store_admin'
     │    └─→ Excel 包含：工号、姓名、性别、电话、所属店面
     │
     ↓
数据入库，examine_state = '待审核'
     │
     │ 2. 总部系统管理员批量审核
     │    └─→ /api/review/batch_approve_and_enable
     │    └─→ 将 examine_state 变更为 '已启用'
     │
     ↓
4S店信息管理员可登录系统
     │
     │ 3. 4S店信息管理员修改个人信息
     │    └─→ 修改后 examine_state 变更为 '待审核'
     │
     │ 4. 提交总部系统管理员审核
     │    └─→ /api/review/batch_approve_and_enable
     │
     ↓
审核通过，examine_state 恢复 '已启用'
```

### 6.2 总部信息员添加流程

```
总部系统管理员
     │
     │ 1. 批量导入总部信息员
     │    └─→ role_type = 'headquarter_info'
     │    └─→ store_id 自动设为 0
     │
     ↓
数据入库，examine_state = '已启用'（直接启用）
     │
     ↓
总部信息员可登录系统
     │
     │ 2. 总部信息员修改个人信息
     │    └─→ 修改后 examine_state 变更为 '待审核'
     │
     │ 3. 提交总部系统管理员审核
     │    └─→ /api/review/batch_approve_and_enable
     │
     ↓
审核通过，examine_state 恢复 '已启用'
```

### 6.3 4S店业务顾问/财务审核添加流程

```
4S店信息管理员（已启用状态）
     │
     │ 1. 批量导入业务顾问/财务审核
     │    └─→ role_type = 'consultant' / 'financial'
     │    └─→ store_id 自动填充为当前用户所属店面
     │
     ↓
数据入库，examine_state = '待审核'
     │
     │ 2. 4S店信息管理员提交审核
     │    └─→ /api/review/batch_approve_and_enable
     │
     ↓
审核通过，examine_state = '已启用'
```

### 6.4 个人信息修改审核流程

```
员工（信息员/业务顾问/财务审核）
     │
     │ 修改个人信息并保存
     │    └─→ 修改后 examine_state = '待审核'
     │
     ↓
总部系统管理员审核页面
     │
     │ 查看待审核列表
     │    └─→ role_type + examine_state = '待审核'
     │
     │ 审核操作
     │    └─→ approve：examine_state = '已启用'
     │    └─→ reject：examine_state = '已拒绝'，数据回滚
     │
     ↓
员工状态更新
     │
     │ examine_state = '已启用'
     │    └─→ 可正常登录使用
     │
     │ examine_state = '已拒绝'
     │    └─→ 可查看拒绝原因，重新修改提交
     │
     ↓
```

### 7.3 ImportExcelService 核心逻辑更新

#### 7.3.1 新增方法：按角色类型处理导入

```java
/**
 * 根据角色类型执行导入
 */
public Map<String, Object> importExcel(MultipartFile file, String roleType, Integer storeId, Integer createBy) {
    List<Map<String, Object>> failedList = new ArrayList<>();
    int successCount = 0;
    int pendingReviewCount = 0;
    
    // 1. 解析 Excel
    Workbook workbook = WorkbookFactory.create(file.getInputStream());
    Sheet sheet = workbook.getSheetAt(0);
    
    // 2. 获取当前登录用户的 store_id
    Integer currentStoreId = getCurrentUserStoreId(createBy);
    
    // 3. 遍历数据行
    for (int i = 1; i <= sheet.getLastRowNum(); i++) {
        Row row = sheet.getRow(i);
        if (row == null) continue;
        
        try {
            Map<String, Object> data = parseRow(row, roleType);
            String error = validateData(data, roleType);
            if (error != null) {
                failedList.add(createFailedItem(i + 1, error));
                continue;
            }
            
            // 4. 根据角色类型处理
            int roleId = 0;
            switch (roleType) {
                case "store_admin":
                    // 4S店信息管理员：根据店面名称匹配 store_id
                    String storeName = (String) data.get("store_name");
                    Integer targetStoreId = storeService.getIdByName(storeName);
                    roleId = infoUserService.insertWithStore(data, targetStoreId, createBy);
                    break;
                    
                case "headquarter_info":
                    // 总部信息员：store_id = 0，状态为已启用
                    roleId = infoUserService.insertAsHeadquarter(data, createBy);
                    break;
                    
                case "consultant":
                    // 业务顾问：store_id 为当前用户所属店面
                    roleId = businessConsultantService.insert(data, currentStoreId, createBy);
                    break;
                    
                case "financial":
                    // 财务审核：store_id 为当前用户所属店面
                    roleId = financialUserService.insert(data, currentStoreId, createBy);
                    break;
                    
                case "member":
                    // 会员：总部管理员可指定店面，4S店管理员只能选本店
                    Integer memberStoreId = determineMemberStoreId(data, currentStoreId, createBy);
                    roleId = ownerUserService.insert(data, memberStoreId, createBy);
                    break;
            }
            
            successCount++;
            // 默认都是待审核状态（除总部信息员外）
            if (!"headquarter_info".equals(roleType)) {
                pendingReviewCount++;
            }
            
        } catch (Exception e) {
            failedList.add(createFailedItem(i + 1, e.getMessage()));
        }
    }
    
    workbook.close();
    
    // 5. 返回结果
    Map<String, Object> result = new HashMap<>();
    result.put("total", successCount + failedList.size());
    result.put("success", successCount);
    result.put("failed", failedList.size());
    result.put("pending_review", pendingReviewCount);
    result.put("failed_list", failedList);
    return result;
}
```

 */
private Integer determineMemberStoreId(Map<String, Object> data, Integer currentStoreId, Integer createBy) {
    String storeName = (String) data.get("store_name");
    if (storeName == null || storeName.isEmpty()) {
        // 未指定店面，使用当前用户所属店面
        return currentStoreId;

    if (!hasPermissionForStore(currentStoreId, targetStoreId)) {

    }
    return targetStoreId;

        int successCount = 0;
        
        for (Integer id : ids) {

                switch (roleType) {

                            infoUserService.approveAndEnable(id);
                        } else {
                            infoUserService.reject(id);
                        }
                        break;
                    case "consultant":
                        if ("approve".equals(action)) {
                            consultantService.approveAndEnable(id);
                        } else {
                            consultantService.reject(id);
                        }
                        break;
                    case "financial":
                        if ("approve".equals(action)) {
                            financialUserService.approveAndEnable(id);
                        } else {
                            financialUserService.reject(id);
                        }
                        break;
                }
                successCount++;
            } catch (Exception e) {
                failedList.add(createFailedItem(id, e.getMessage()));

        Map<String, Object> result = new HashMap<>();
        result.put("total", ids.size());
        result.put("success", successCount);
        result.put("failed", failedList.size());
        result.put("failed_list", failedList);

## 十三、风险与注意事项

### 13.1 潜在风险

| 风险点 | 影响 | 应对措施 |
|--------|------|----------|
| 权限校验遗漏 | 数据越权访问 | 前后端双重校验，后端必须校验 |
| 导入时 store_id 填充错误 | 人员归属错误 | 使用事务，导入失败回滚 |
| 批量审核时状态判断错误 | 误操作 | 明确状态流转规则，仅特定状态可操作 |
| 并发审核同一记录 | 数据不一致 | 使用乐观锁或悲观锁 |
| Excel 模板字段与代码不一致 | 导入失败 | 维护字段映射配置 |

### 13.2 注意事项

1. **store_id 填充规则**：
   - 总部管理员导入 `store_admin`：根据 Excel 中的店面名称匹配
   - 总部管理员导入 `headquarter_info`：自动设为 0
   - 4S店管理员导入：自动填充当前用户的 store_id

2. **权限校验**：
   - 所有接口后端必须校验权限
   - 4S店管理员只能操作本店（store_id 相同）的人员
   - 审核功能仅总部管理员可用

3. **状态流转**：
   - `待审核` → `已通过`（审核通过）
   - `待审核` → `已拒绝`（审核拒绝）
   - `已通过` → `已启用`（手动启用）
   - `待审核` → `已启用`（批量审核并启用）

4. **事务管理**：
   - 批量导入使用事务，保证数据一致性
   - 批量审核使用事务，保证状态一致性

5. **日志记录**：
   - 记录导入、审核、启用的详细操作日志
   - 记录操作用户、操作时间、操作内容
